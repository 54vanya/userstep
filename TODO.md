## Пустые сегменты длинных нот (".")

В UCS длинная нота может проходить сквозь строки/блоки, где в исходном файле стоит
`.` (пустой сегмент), а не `H`. Парсер теперь «протягивает» холд сквозь такие пустые
блоки-распорки (`src/services/ucsParser.ts` → `rowsToNotes`, ветка `lastActive < 0`),
чтобы холд не рвался (см. кейс CS241, блоки #25+).

Нужно:
- **Не учитывать пустые сегменты в общем количестве нот.** Сейчас счётчик хитов
  (`src/utils/noteCount.ts` → `computeHitTimes`) считает КАЖДУЮ строку, занятую холдом
  (`note.row…endRow`), как хит. Для «протянутых» сквозь `.` строк это неверно — такие
  строки в исходнике пусты и не должны идти в `total` / `passed` (тулбар + оверлей
  `NoteCounterOverlay`). Требуется помечать, какие строки холда реальны (`M`/`H`/`W`),
  а какие — мост через `.`, и исключать мостовые из подсчёта.
- Решить, как такие сегменты вести себя при экспорте обратно в `.ucs`
  (`src/services/ucsSerializer.ts`) — сохранять `.` или заполнять `H`.

### Возможные значения сегмента (по CS241)

Сегмент = ячейка в колонке холда внутри пролёта `row…endRow`. По данным CS241
(Mode=Double, CRLF; символы в гриде: `.` `X` `H` `M` `W`; `M`/`W` сбалансированы 53/53):

| Значение | Смысл | Ограничения |
|---|---|---|
| `M` | голова холда | ровно одна, первая строка пролёта |
| `H` | тело | 0..N |
| `W` | хвост холда | ровно один, последняя строка пролёта |
| `.` | пустой сегмент / «мост» | есть в исходнике, **не считается** в общем кол-ве |

`X` сегментом холда быть не может — это отдельный tap (в другой колонке). Между `M`
и `W` встречаются только `H` и `.`. Пропуски `.` бывают и внутри блока, и через целые
пустые блоки-распорки (кросс-блочный мост, `continued`/`continues`).

## Параметры BPM-сегмента (блока) — реальные диапазоны

По CS241 (110 блоков):

| Параметр | Значения в CS241 | Тип/диапазон |
|---|---|---|
| BPM | 0.1 … 292 (`0.1, 2.125, 5.71875, 8.5, 11.4375, 22.875, 31.5, 34, 36.5, 68, 104, …, 272, 292`) | дробное, может быть **< 1**; в CS355 есть 900 |
| Delay | 0, 120 | целое мс ≥ 0 |
| Beat | 4 | целое (формат допускает 2/3/6/8…) |
| Split | 1, 2, 3, 4, 8, 16, 32, 64, 128 | целое (степени 2 + 3) |

Гиммик-приём: чередование быстрых `Split=1`/высокий BPM и медленных
`Split=128`/низкий BPM имитирует scroll-speed/stop.

(парсер `ucsParser.ts`, `blockRowCount` округляет, `updateBlock` не ломает неполные
блоки). В `BlockSettingsPopup`: BPM `min=0.001 step=any`, Split-список расширен до 128
